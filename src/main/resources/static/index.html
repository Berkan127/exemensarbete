Prompt 2 - Mikrotjanstbaserat system

En enkel hemsida som sammanfattar kraven och utvecklingsstegen for GitHub-uppladdning.

Java 17 Spring Boot PostgreSQL Docker API Gateway

Syfte

Jämföra monolit och mikrotjanster med fokus pa skalbarhet, distribution, komplexitet och beteende under hog belastning.

Arkitektur

  • Tre mikrotjanster: User Service, Product Service, Order Service.
  • Varje tjanst har egen kodbas, egen PostgreSQL-databas och separat deployment.
  • API Gateway ar enda entry point och routar trafik till ratt tjanst.
  • Endast synkron REST-kommunikation (ingen message broker).

Del 1 - Struktur

  • Skapa User, Product, Order och API Gateway.
  • Konfigurera beroenden och Docker-grund.

Del 2 - User Service

  • Datamodell + egen databas.
  • POST /users
  • GET /users/{id}

Del 3 - Product Service

  • Datamodell + egen databas.
  • GET /products, POST /products, GET /products/{id}
  • Forbered lagerhantering.

Del 4 - Order Service

  • Datamodell + egen databas.
  • POST /orders, GET /orders/{id}, GET /orders/user/{userId}
  • Affarslogik i Order Service.

Del 5 - Kommunikation

  • Order Service anropar User och Product via REST.
  • Hamta user, hamta produkter, kontrollera/uppdatera lager.
  • Berakna totalpris och hantera externa fel.

Del 6 - API Gateway

  • Konfigurera routing till alla tjanster.
  • Testa hela flodet via gatewayn.

Del 7 - Logging/Felhantering

  • Logga interna handelser och inkommande anrop.
  • Logga inter-service anrop.
  • Hantera timeout och nere-tjanst med tydliga fel.

Del 8 - Docker lokalt

  • Dockerisera alla tjanster och databaser.
  • Kora lokalt via Docker Compose.

Del 9 - AWS

  • Deploya varje tjanst separat via ECS eller EC2.
  • Verifiera distribuerad drift.

Del 10 - Skalning/Prestanda

  • Skala endast Order Service.
  • Mat latency, throughput, CPU, memory och inter-service latency.

Del 11 - Jamforelse med monolit

  • Analysera skillnader i skalbarhet, deployment-komplexitet och hoglastbeteende.

Tips: Gor en commit per del (Del 1 till Del 11) for tydlig GitHub-historik.
